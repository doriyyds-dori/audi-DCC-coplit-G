# 智慧外呼助手第一阶段角色与权限范围说明

> 日期：2026-03-24

---

## 1. 第一阶段最小角色集合

| 角色 | 存在理由 |
|------|---------|
| **超级管理员（super_admin）** | 管理所有门店和用户，查看全局统计数据 |
| **普通用户（user）** | 一线销售顾问，使用话术助手并产生通话记录 |

**只需 2 个角色。** 第一阶段不设门店管理员、审批角色或只读角色。

---

## 2. 每个角色可见范围

| 角色 | 可见页面 | 可见数据 |
|------|---------|---------|
| 超级管理员 | 话术助手页、管理后台（用户管理 + 统计） | 所有门店、所有用户、所有通话记录 |
| 普通用户 | 话术助手页 | 仅自己的通话记录（当前阶段不展示） |

---

## 3. 每个角色可执行动作

### 超级管理员

| 动作 | 说明 |
|------|------|
| 登录/登出 | — |
| 使用话术助手 | 与普通用户一致 |
| 新增用户 | 指定用户名、密码、所属门店 |
| 禁用/启用用户 | 禁用后该用户无法登录 |
| 新增/编辑门店 | 修改门店名称 |
| 查看统计数据 | 按日期、门店、用户筛选通话记录 |

### 普通用户

| 动作 | 说明 |
|------|------|
| 登录/登出 | — |
| 使用话术助手 | 通话模拟、FAQ 查询、CSV 上传 |
| 标记通话状态 | 每次通话结束后选择：未接通/无意向/有意向 |

---

## 4. 第一阶段明确不做的权限能力

| 不做 | 原因 |
|------|------|
| 门店管理员角色 | 400 人规模用一个超级管理员足够，不需要分级管理 |
| 多级组织架构 | 第一阶段只有"门店"一层，不做区域/大区 |
| 细粒度字段权限 | 不做"某角色只能看某些字段" |
| 审批流程 | 不做用户注册审批、话术修改审批 |
| 普通用户查看自身统计 | 第一阶段统计仅管理端可见 |
| 操作日志/审计 | 第一阶段不记录管理操作 |

---

## 5. 推荐的最小后台菜单

仅超级管理员可见，共 3 个菜单项：

```
管理后台
├── 今日统计    ← 默认首页，按门店/用户显示当日通话数据
├── 用户管理    ← 用户列表、新增、禁用
└── 门店管理    ← 门店列表、新增、编辑
```

不做：系统设置、权限配置、话术在线编辑、报表中心、消息通知。

---

## 6. 角色与后续数据设计的关系

### users 表

- `role` 字段区分超级管理员和普通用户，取值为 `super_admin` 或 `user`
- `store_id` 字段关联门店，决定该用户归属于哪个门店
- `enabled` 字段控制是否可登录

### 通话记录归属

- `call_records.user_id` 关联到 `users.id`，记录是哪个普通用户产生的
- 超级管理员如果也使用话术助手，其通话记录同样写入，但可在统计中按需排除

### 统计范围

- 超级管理员查看统计时，可筛选全部门店或指定门店
- 普通用户第一阶段不查看任何统计（后续阶段可考虑开放个人统计）

---

## 7. 第一阶段角色设计结论

1. **只做 2 个角色**：超级管理员和普通用户，不设中间层
2. **权限划分极简**：普通用户只能用话术助手和标记通话状态，管理员额外可管理用户和查看统计
3. **管理后台只有 3 个页面**：今日统计、用户管理、门店管理
4. **角色通过 users 表的 role 字段区分**，不需要独立的权限表
5. **第一阶段不做权限扩展预留**，等真正需要时再加
