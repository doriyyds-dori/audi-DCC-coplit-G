# 门店管理员与双层话术体系设计方案

> 日期：2026-03-26

---

## 1. 本轮目标

**在现有两角色体系基础上新增"门店人员管理员"角色，建立"总部统一话术 + 门店自定义话术"双层话术体系，并定义门店级数据权限边界。**

---

## 2. 角色模型

### 超级管理员（super_admin）

| 维度 | 内容 |
|------|------|
| **核心职责** | 管理全局用户、门店、统一话术，查看全局业务数据 |
| **可见范围** | 所有门店、所有用户、所有通话记录、所有话术 |
| **可操作范围** | 用户 CRUD、门店 CRUD、统一话术编辑、全局统计查看 |
| **不能做** | 无限制（最高权限） |

### 门店人员管理员（store_admin）

| 维度 | 内容 |
|------|------|
| **核心职责** | 管理本门店员工和门店自定义话术，查看本店业务数据 |
| **可见范围** | 本门店用户、本门店通话记录、统一话术（只读）、本店自定义话术 |
| **可操作范围** | 本店用户新增/编辑/禁用、本店自定义话术编辑、本店统计查看 |
| **不能做** | 不能看其他门店数据、不能改统一话术、不能管理其他门店用户、不能改自己的角色 |

### 普通使用者（user）

| 维度 | 内容 |
|------|------|
| **核心职责** | 使用话术助手完成外呼，标记通话状态 |
| **可见范围** | 话术助手页（统一话术 或 本店自定义话术） |
| **可操作范围** | 选择话术来源、通话模拟、FAQ 查询、标记通话状态 |
| **不能做** | 不能进入管理后台、不能编辑话术、不能查看统计数据 |

---

## 3. 数据权限边界

### 权限矩阵

| 数据 | 超级管理员 | 门店人员管理员 | 普通使用者 |
|------|-----------|--------------|-----------|
| 全局用户列表 | ✅ 全部 | ❌ 仅本店 | ❌ |
| 全局门店列表 | ✅ 全部 | ❌ 仅本店信息 | ❌ |
| 通话记录 | ✅ 全部门店 | ✅ 仅本店 | ❌ |
| 统一话术 | ✅ 可编辑 | ✅ 只读 | ✅ 只读 |
| 门店自定义话术 | ✅ 全部门店 | ✅ 仅本店可编辑 | ✅ 仅本店只读 |

### 后端强制收口原则

- **所有管理接口必须根据当前用户的 role 和 storeId 做过滤**
- 门店人员管理员调用用户列表 → 后端只返回 `WHERE storeId = 当前用户storeId`
- 门店人员管理员调用统计 → 后端只返回 `WHERE storeId = 当前用户storeId`
- 不能只靠前端隐藏菜单或按钮来实现权限控制

## 4. 双层话术体系

### 概念定义

| 层级 | 名称 | 维护者 | 归属 |
|------|------|--------|------|
| 总部层 | 统一话术 | 超级管理员 | 全局可用 |
| 门店层 | 门店自定义话术 | 门店人员管理员 | 仅本店可用 |

### 继承规则

1. **初始创建**：门店人员管理员点击"基于统一话术创建自定义话术"时，系统将当前统一话术完整复制一份到该门店名下
2. **复制后独立**：门店自定义话术与统一话术完全解耦，后续独立维护
3. **不自动同步**：超级管理员后续修改统一话术时，已有的门店自定义话术不会被覆盖或合并
4. **可重新创建**：如门店想重新同步，可以删除当前自定义话术后重新基于最新统一话术创建

### 话术数据结构方向

- 每套话术需要有唯一标识
- 需要区分 `scope`：`global`（统一话术）或 `store`（门店自定义话术）
- 门店自定义话术需要关联 `storeId`
- 需要记录 `sourceScriptId`（来源统一话术 ID，用于溯源，不用于同步）

## 5. 使用者的话术选择逻辑

### 通话前最小流程

```
用户进入话术助手页
   ↓
选择客户场景（保客 / 新客）
   ↓
系统检查本店是否有自定义话术
   ├── 没有 → 直接使用统一话术，不显示选择器
   └── 有   → 显示选择器：
              ├── 统一话术（总部版）
              └── 门店自定义话术（本店版）
   ↓
用户选择后进入通话模拟
```

### 规则说明

- 如果本门店没有自定义话术 → 只展示统一话术，无需选择
- 如果本门店已有自定义话术 → 允许二选一
- 选择结果在通话记录中需要记录（见第 7 节）
- 超级管理员使用话术助手时，按其 storeId（如有）决定可选范围；如无 storeId 则只能用统一话术

## 6. 业务分析权限

| 角色 | 分析能力 |
|------|---------|
| 超级管理员 | 可查看全局统计 + 按门店筛选查看单店统计 |
| 门店人员管理员 | 只能查看本店统计，后端按 storeId 限制 |
| 普通使用者 | **第一版不开放分析能力** |

### 普通使用者不开放分析的理由

- 当前核心场景是管理层监控外呼效率，普通用户查看统计不是刚需
- 开放后需要处理个人维度筛选、隐私边界等额外问题
- 建议在后续版本中按需开放"个人通话统计"

---

## 7. 关键技术原则

### 1. 通话记录必须记录话术来源

通话记录新增字段：

| 字段 | 值 | 说明 |
|------|------|------|
| scriptSource | `global` | 使用了统一话术 |
| scriptSource | `store` | 使用了门店自定义话术 |
| scriptId | TEXT | 具体话术的 ID |

这样后续可以分析：哪种话术的成功率更高。

### 2. 门店级数据权限必须在后端 storeId 级别收口

- 门店人员管理员的所有查询请求，后端从 session 取 storeId，不接受前端传入
- 超级管理员可选传 storeId 做筛选，不传则返回全部
- 这是安全红线，不可退让

---

## 8. 明确暂不做的内容

| 暂不做 | 原因 |
|--------|------|
| 统一话术更新后自动同步门店自定义话术 | 增加巨大复杂度，冲突处理难解决 |
| 多级门店组织架构（区域/大区） | 当前只有门店一层 |
| 跨门店管理员 | 一个门店管理员只管一个店 |
| 复杂审批流 | 话术创建/修改不需要审批 |
| 话术版本合并/diff | 不做版本对比、合并 |
| 大而全分析平台 | 只做最小统计 |
| 话术在线编辑器 | 第一版话术仍通过 CSV 管理 |
| 普通用户查看个人统计 | 后续按需开放 |

---

## 9. 建议的下一步设计顺序

1. **话术数据模型设计** — 定义 scripts 表结构，明确 scope / storeId / sourceScriptId 等字段，以及话术内容的存储形式（CSV 继续 vs 结构化）
2. **store_admin 角色接入** — 在 users 表 role 字段增加 `store_admin` 值，前后端路由守卫适配三角色
3. **后端管理接口门店级收口** — 为门店人员管理员的用户列表、统计查询等接口增加 storeId 过滤
4. **话术选择器 UI** — 在话术助手页增加"话术来源"选择，通话记录增加 scriptSource 字段
5. **门店自定义话术管理页** — 门店管理员可以"基于统一话术创建自定义话术"并编辑
